# JomBoss Mastermind Roadmap

## 1. Purpose of This Roadmap

This roadmap gives Codex the full operating context for the JomBoss Mastermind and data room project. It explains what is being built, why it matters, what has already been created, and how future work should proceed.

This is a living Markdown file. It should be updated as new facts, documents, legal advice, investor feedback, Hult Prize requirements, grant requirements, or CEO instructions become available.

## 2. What We Are Building

JomBoss is building a Codex-powered internal company knowledge system and investor data room. This is not merely a storage folder. It is intended to become the JomBoss Mastermind: an internal operating brain that tracks company truth, evidence, due diligence gaps, legal risks, financial proof, IP ownership, founder governance, client proof, platform documentation, and weekly CEO priorities.

The system should support:

- YC-level due diligence readiness
- Hult Prize, grant, and accelerator evidence readiness
- founder governance cleanup
- IP ownership tracking
- financial and traction verification
- client and project proof
- product and platform documentation
- weekly CEO PA workflow
- investor-safe exports

## 3. Operating Philosophy

- Markdown files are the live source of truth.
- PDFs are dated snapshots.
- DOCX files are for legal review and formal signing preparation.
- Signed PDFs are the final executed legal record.
- Every claim should be source-backed.
- CEO instructions may be recorded, but must be labelled as CEO instruction until evidence is uploaded.
- Sensitive files must stay restricted.
- Final legal or investor-facing documents should not be created before source evidence is ready.
- Proposed founder allocation must not be treated as legal shareholding.

## 4. Current Stage

The repository is currently in the foundation-building stage.

The current work is not to finalise every document. The current work is to:

- create system rules
- create metadata standards
- create trackers
- record current truth
- separate legal facts from proposed positions
- identify missing evidence
- prepare to finalise documents one by one later

## 8. Working Loop Between G, ChatGPT and Codex

G provides decisions, facts, evidence, and final approval.

ChatGPT helps reason, stress-test, draft, and review.

Codex stores, audits, updates, and maintains the repository.

Codex should not decide legal, equity, or investor matters on its own.

The working loop is:

1. Codex audits one section.
2. G discusses missing or uncertain items with ChatGPT.
3. G confirms facts or decisions.
4. Codex updates repo documents.
5. Trackers and open issues are updated.
6. Repeat section by section.

## 9. What Codex Should Not Do Yet

- Do not create final founder agreement yet.
- Do not create final investor-safe cap table yet.
- Do not mark proposed beneficial allocation as legal shareholding.
- Do not create public or investor-safe exports from legal-sensitive files.
- Do not push sensitive files to GitHub before GitHub safety plan is approved.
- Do not reproduce personal data in investor-safe summaries.
- Do not delete Markdown source files after export.
- Do not overwrite signed documents.

## 10. GitHub and Version Control Direction

GitHub setup should be planned carefully because the repository contains legal-sensitive, personal-data, financial-sensitive, and restricted internal documents.

GitHub must be private if used.

Before any GitHub push:

- inspect current Git status
- create `.gitignore`
- identify local-only restricted files
- decide what can be safely tracked
- avoid pushing personal data, bank records, signed agreements, or client-sensitive evidence unless explicitly approved
- get G's approval before pushing

The current GitHub target discussed by G is:

`zher-g/jomboss`

This target should not be pushed to until the GitHub safety plan is approved.

## 11. Definition of YC-Level Readiness

YC-level readiness means the system is:

- accurate
- evidence-backed
- source-traceable
- dated
- version-controlled
- confidentiality-classified
- investor-safe when exported
- honest about missing evidence
- not overclaiming
- able to withstand due diligence questions

## 12. Current Stop Rule

Codex should stop and ask G before:

- drafting final legal documents
- exporting investor-safe PDFs
- moving sensitive folders
- pushing to GitHub
- changing cap table or founder allocation status
- deleting or archiving source evidence
- treating any unresolved issue as solved
